Gorilla paper → startup map

Old boring APIs, rebuilt for agents.

A strategy memo for finding devtool startup ideas: take legacy APIs with ugly docs/state machines, make them OSS + agent-native, and become the thing LLMs recommend by default.

Thesis: LLMs recommend what is easy to retrieve, easy to explain, and safe to call. Most old APIs fail all three.
Wedge: OpenAPI + MCP + clean SDKs + examples + validation beats “prompt GPT with a cursed docs page.”

1. Gorilla paper summary

The key lesson from Gorilla: Large Language Model Connected with Massive APIs.

Tool-use is not a prompt problem. It is a retrieval + schema + validation problem.
Core insight

Fine-tune or teach models to use retrieved docs, not merely read them.

Gorilla trains a LLaMA-based model to map natural-language tasks to valid API calls, using retriever-aware training so it can adapt to changing API docs.

  • Builds APIBench: ~1,645 ML APIs from HuggingFace, TorchHub, TensorFlow Hub.
  • Measures accuracy, wrong API errors, and hallucinated APIs separately.
  • Uses AST matching to verify whether generated API calls are structurally valid.
Problem

LLMs hallucinate tools

They invent API names, wrong packages, fake arguments, and stale usage patterns.

Solution

Retriever-Aware Training

Train the model to use relevant retrieved docs and ignore irrelevant docs.

Startup lesson

Win the retrieval layer

If your docs/specs are cleaner than legacy providers, agents will route through you.

2. Underrated places to mine ideas

Look for demand hiding inside boring integration pain.

Directory

APIs.guru OpenAPI Directory

14k+ OpenAPI specs. Search for ugly but high-value domains: tax, shipping, CRM, KYC, payroll, healthcare, government data.

Demand signal

Postman Public Network

Collections with lots of forks but bad SDKs/docs indicate “people need this but hate integrating it.”

Marketplace

RapidAPI categories

Address validation, email/phone verification, OCR, weather, tracking, tax/VAT, enrichment, IP intelligence.

Automation

Zapier / Make / n8n directories

If non-devs automate it repeatedly, devs will want an agent-native programmable primitive.

Bureaucracy

Government APIs

USPS, SEC EDGAR, IRS charity lookup, sanctions lists, court data, customs/tariff APIs: boring, high-intent, usually painful.

GitHub rot

Abandoned SDKs

Search for unofficial or deprecated SDKs. If usage continues after maintenance dies, demand is wearing a corpse costume.

Pain clusters

StackOverflow / GitHub issues

Repeated questions like OAuth refresh, webhook verification, tax API errors, address normalization = startup smell.

Compliance

Regulated workflow APIs

KYC, KYB, AML, payout routing, invoice compliance. Agents need guardrails, audit logs, and deterministic validation.

5. How to become the LLM default

Make the agent’s job embarrassingly easy.

OpenAPI first

Clean spec, examples, error schemas, typed objects.

MCP server

Expose safe tools with descriptive names and constraints.

OSS SDKs

JS + Python, tiny surface area, great README.

Agent docs

llms.txt, AGENTS.md, copy-paste task examples.

Validation

Dry-run, confidence, audit logs, reversible operations.

Documentation SEO for agents

  • Comparison pages: “USPS API for AI agents”, “Shippo alternative for agents”.
  • Natural language examples: “send a payout”, “validate this address”, “replay failed webhook”.
  • Every API page includes request, response, failure modes, retry policy.
  • Publish test fixtures and sandbox keys/mocks.
  • Use package names that match intent: agent-address, agent-webhooks.
  • Ship a CLI: agents love shellable primitives.
